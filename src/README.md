## Зауваження та код:

### 1. Назва класу має бути з великої літери {#зауваження-1}
У Java класи зазвичай пишуть у `PascalCase`.

```java
class humanIMB { // як було
class HumanIMB { // виправлено
```

### 2. Атрибути класу мають бути `private` {#зауваження-2}
Поля W та H мають бути private, щоб забезпечити інкапсуляцію.

```java
public double W; // як було
private double W; // виправлено
```

### 3. imb не повинен бути static {#зауваження-3}
Оскільки imb залежить від кожного окремого екземпляра класу, він не має бути static.

```java
private static double imb; // як було
private double imb; // виправлено
```

### 4. Обчислення imb варто винести в окремий метод {#зауваження-4}
Логіку розрахунку imb слід винести в окремий метод calculateBMI(), бо вона часто повторюється у коді.
Через це сам imb тепер зайвий (його можна обчислити одразу, тому не треба зберігати як окрему змінну).

```java
imb = W / (H * H); // як було (в конструкторі)

private double calculateBMI() { // правильно
    return weight / (height * height);
}
```

### 5. Методи мають відповідати Java-конвенціям {#зауваження-5}
Java використовує гетери/сетери у форматі getSomething()/setSomething(value).

```java
public double takeW() { return W; } // як було
public double getWeight() { return weight; } // виправлено
```

### 6. Не варто обчислювати imb у сетерах/конструкторах {#зауваження-6}
Сетери мають змінювати лише значення змінних, а розрахунок imb слід робити у відповідному методі для обчислення значення - calculateBMI(). Тим більше не варто обчислювати ІМТ прямо в конструкторі, адже це різні зони відповідальності (конструктор точно не повинен таким займатись, якщо у класі є відповідний метод для цього)

```java
public void setWeight(double weight) {
    this.weight = weight;
    imb = weight / (height * height); // неправильно
}
```

### 7. Метод takeImt() має бути getBMI() {#зауваження-7}
Назви методів мають зрозуміло відображити їх зону відповідальності  

```java
public static double takeImt() { return imb; } // неправильно
public double getBMI() { /* ... */ } // правильно
```

### 8. Метод Result() не повинен бути static {#зауваження-8}
Метод Result() використовує нестатичну змінну imb, тому він також має бути нестатичним.

```java
public static String Result() { // неправильно
public String getResult() { // правильно
```

### 9. Змінна string у Result() зайва {#зауваження-9}
Можна одразу повертати рядки без зайвої змінної.

```java
String string = null; // зайва змінна
if (imb >= 18.5 && imb < 25) return "Norm"; // одразу повертаємо значення
```

### 10. Використання && замість & {#зауваження-10}
У логічних виразах у Java використовують &&, а & застосовують у побітових операціях.

```java
if (imb >= 18.5 & imb < 25) // неправильно
if (imb >= 18.5 && imb < 25) // правильно
```

### 11. Варто використовувати if-else разом із return {#зауваження-11}
Це зменшить кількість зайвих перевірок та зробить умовний оператор лаконічнішим.

```java
// як було
if (imb >=18.5 & imb <25) {
    string ="Norm";
}
if (imb >=25 & imb <30) {
    string ="Warning! ";
}
if (imb >=30) {
    string ="Fat";
}
if (imb <18.5) {
    string ="Deficit";
}

// правильно
if (imb >= 18.5 && imb < 25) return "Norm"; 
if (imb >= 25 && imb < 30) return "Warning!"; 
if (imb >= 30) return "Fat"; 
return "Deficit"; 
```

### 12. У сетерах/конструкторі пропущене ключове слово this {#зауваження-12}
Це важливий момент під для кращого визначення контексту (this.weight - поле класу та weight як аргумент конструктора; this.height - поле класу та height як аргумент сетера)

```java
// як було
public humanIMB(double w, double h) {
    W = w;
    H = h;
    imb = W / (H * H);
}

// правильно
public HumanBMI(double weight, double height) {
    this.weight = weight;
    this.height = height;
}
```


### 13. Варто використовувати сетери в конструкторі {#зауваження-13}
Це дозволить продумати логіку валідації та перевикористати її у конструкторі класу

```java
// як було
public humanIMB(double w, double h) {
    W = w;
    H = h;
    imb = W / (H * H);
}

// правильно
public HumanBMI(double weight, double height) {
    setWeight(weight);
    setHeight(height);
}
```

### 14. Клас краще переносити у окремий файл {#зауваження-14}
Клас для визначення ІМТ краще перенести у окремий файлик, для кращого розподілу логіки та відповідальності між частинами проєкту

### 15. Не треба забувати про модифікатори доступу для класу {#зауваження-15}
Завжди краще уточнити рівень доступу до класу, для кращої читабельності та зрозумілості коду

### 16. Варто слідкувати за логічністю назв атрибутів та аргументів {#зауваження-16}
Краще написати декілька слів, але щоб назва змінної (аргумента або атрибута) була зрозумілою

```java
// як було
public humanIMB(double w, double h) {
    // ...
}

// правильно

    // ...
}
```

### 18. Слід дотримуватись єдиного форматування {#зауваження-18}
Варто слідкувати за пробілами - краще дотримуватись єдиного формату коду

```java
// як було
if (imb >=25 & imb <30) {
    string ="Warning!";
}

// як правильно
if (imb >= 25 & imb < 30) {
    string = "Warning!";
}
```

### 19. Потрібно слідкувати за зрозумілістю повідомлень {#зауваження-18}
У коді не має бути неоднозначних або нерелевантних повідомлень

```java
// як було
if (imb >=25 & imb <30) {
    string ="Warning! ";
}

// як правильно - логічне повідомлення, яке вписується в контекст
if (imb >= 25 & imb < 30) {
    string ="Overweight! ";
}
```
